## Dev setup

### Prerequisites

- Python 3.10+
- Node.js 18+
- Git

### Steps

```bash
# 1. Fork on GitHub, then clone your fork
git clone https://github.com/YOUR_USERNAME/hemogrid.git
cd hemogrid

# 2. Install Python backend dependencies
pip install -r requirements.txt

# 3. Install frontend dependencies
cd frontend && npm install && cd ..

# 4. Confirm everything starts
python main.py
# Expected: "[PASS] Startup assert: 536 available bags in baseline CSV."

# 5. In a second terminal, start the frontend dev server
cd frontend && npm run dev
# Dashboard at http://localhost:5173
```

## Invariants

The dataset and matching engine enforce eight hard invariants. Violating any of them will cause a loud `AssertionError` — not a silent wrong answer.

1. **Inventory is a derived query — never a stored column or table.**
   `available = bags WHERE status='available' AND expiry_date >= TODAY`

2. **Bags carry ABO/Rh only.** Extended phenotype (14 antigens) lives on the donor row, not the bag row.

3. **Matcher uses real GPS.** `latitude`/`longitude` are precise coordinates. `display_latitude`/`display_longitude` are map-only jitter — never pass them to the routing engine.

4. **Antibody safety is universal.** An antigen-positive unit is never issued against any antibody — allo or auto, active or historical. Auto-antibody patients get `requires_adsorption_workup=True` and route directly to G3; they can never be auto-issued.

5. **Untyped donor + immunized patient = excluded.** Phenotype cannot be proven safe, so the unit is blocked for immunized patients. Allowed (flagged) for non-immunized patients.

6. **Primary keys are unique; zero orphaned foreign keys** across all eight tables (bags→donors, bags→banks∪facilities, patients→facilities, donors→banks, antibodies→patients, reservations→patients/donors/bags).

7. **`expiry_date = collection_date + 42`** (42-day packed-RBC shelf life). `max(collection_date) ≤ TODAY − 1`. `status=available` implies `expiry_date ≥ TODAY`.

8. **Emails and phone numbers are format-valid and globally unique.** Phones match `^\+91[6-9]\d{9}$`. No `\x` escape sequences in any ID field.

If you break any of these, the asserts in `engine/state.py` and `engine/match.py` will fire. Fix the root cause; do not silence the assert.

## Project conventions

### Python

- Config knobs go at the top of every module under a `# ── Config ──` comment block.
- Asserts are **loud** — they `raise`, never swallow. If an assert fires, it is a bug to fix, not a warning to suppress.
- `RANDOM_SEED=42` and `TODAY=2026-06-06` are fixed across all scripts. The dataset is intentionally forward-dated for demo purposes.
- Use Pandas for any data manipulation. Keep the engine layer free of HTTP concerns.

### Build pipeline

- **Never manually edit any file in `data/build/`** — these CSVs are the canonical dataset, generated by the build pipeline. Changes you make by hand will be overwritten the next time someone runs the build.
- To change the dataset, modify `build_datasets.py` (or `build_jitter.py` / `build_map.py` for the map layer), then re-run the full pipeline:

  ```bash
  python build_datasets.py
  python build_jitter.py
  python build_map.py
  ```

### Reports

- Build scripts always open report files with `'w'` (overwrite), stamp a `RUN_ID` on line 1, then re-read the file and assert the on-disk `RUN_ID` matches the in-memory value.
- Never hand-type counts or statistics into a report — regenerate all numbers from live dataframes.

### Dead code

- `backend/main.py` is a superseded stub — the live gateway is the root `main.py`. Do not add to it.

## Modifying the dataset

```bash
# Full rebuild — deterministic, takes several minutes
python build_datasets.py   # Phase 1: extract real data + fill synthetic gaps
python build_jitter.py     # Phase 2: add privacy-jittered display coords
python build_map.py        # Phase 3: generate Leaflet map data
```

After rebuilding, run the full verification suite (see below) before opening a PR.

## Verifying your changes

There is no automated test suite (pytest / jest). Correctness is enforced by loud assertion guards in the engine. Run these before every PR:

```bash
# 1. Match all 164 patients — prints G1/G2/G3 tier counts and the top candidate for one sample patient
python -m engine.match

# 2. Full state mutation test: reset → issue → donate → all assertion guards
python -m engine.state --test-flow

# 3. Routing assertion checks
python -m engine.routing --self-test
```

All three must complete without an `AssertionError` or `[FAIL]` line.

For frontend changes, manually test in the browser — start both servers and exercise the golden path: select a patient, view match tiers, click "Issue Unit", verify the activity log updates, and reset the simulation.

## Branch and PR conventions

| Prefix | Use for |
|--------|---------|
| `feature/` | New functionality |
| `fix/` | Bug fixes |
| `refactor/` | Code cleanup with no behaviour change |
| `data/` | Dataset changes (requires full pipeline re-run) |

- Keep PRs focused — one concern per PR. A dataset change and an engine change are two separate PRs.
- PR title: imperative, under 70 characters. ("Add Kell antigen to G1 concordance scoring", not "Fixed things")
- All three engine self-tests must pass.
- No `\x` escape sequences in any generated ID.
- No hand-typed constants in report output — all numbers must be regenerated from live dataframes.
- `data/working_sim.db` is excluded from the repo (it is the mutable simulation state). Do not commit it.

## Reporting bugs

Use [GitHub Issues](../../issues/new?template=bug_report.md). Please include:

- What you expected to happen
- What actually happened (full traceback if applicable)
- Steps to reproduce
- Output of `python -m engine.state --test-flow` if the issue is in the engine layer
